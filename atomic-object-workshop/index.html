Atomic Object Workshop

A simulated Delivery Lead workshop for Caring Health — a story-mapping session I facilitated, then handed to Claude Code to turn the agreed beta scope into a clickable prototype. This page collects the prompt, the board, the build, and the result.

Role
Facilitator (Delivery Lead candidate)
Format
~2-hour product backbone / journey-map workshop
Then
Beta scope → clickable prototype, built with Claude Code

Atomic Object, a software consultancy, ran a final-round interview as a simulated workshop: they handed over a client brief and I had an hour to prepare and then facilitate a product backbone exercise with a designer, a developer, and two delivery leads. The goal was to take a large, fuzzy list of features and organize it into a prioritized backbone the dev team could actually start from.

The client was Caring Health, a HIPAA-compliant app that lets a caregiver request an at-home visit from a licensed nurse for a loved one — vitals, wound care, medication help, a home-safety check. The beta was scoped to three things: set up & login, request & confirm a service, and track the nurse's arrival. In my prep hour I set up a journey-map template in Miro and a basic flow for how we'd map features against it.

During the session we sorted features into rows — green "must have" stickies for the beta, yellow "should/could have" below, and a red "nopes" parking lot — across the activity backbone. Afterward, out of curiosity, I wanted to see what the agreed beta actually felt like as a flow. So I gave Claude Code the board and the brief and asked it to build a clickable prototype from the green must-haves only. Everything below is that artifact, start to finish.

The prompt

My first instinct was a loose, conversational ask. Before running it, I refined it into something a coding agent could execute against without guessing: clear inputs, a concrete definition of done, and an explicit instruction to surface its assumptions instead of silently inventing the parts we'd flagged as open. Both versions are below.

What I built

caring-health-prototype.html — one self-contained file (HTML/CSS/JS, no build step, no external assets). I rendered all 41 board slices to reconstruct the journey map, read the brief and Dana's persona, and pulled scope only from the green "must have" stickies across the three activity bands. Named brainstorm flows, yellow should/could, and red "nopes" were excluded.

I tap-tested the whole path in a browser: Dana signs up → creates password → enters caregiver info → enters patient (Walter) info → home → request a visit → pick services → location → day/time → review → confirm → live nurse tracking → back home with the visit logged. No dead-ends; the core loop closes.

Must-haves covered:

  • Set Up & Onboarding — email signup, password, caregiver name+phone, patient name/DOB/relationship/address, support number (in menu), logout
  • Request & Confirm — select one+ services, choose patient, choose/confirm location, pick day + time, confirmation via notification, cancel option
  • Track Nurse — "on the way" status, "arrived" notification, "completed/left" notification (simulated push toasts + a live timeline)

Where we guessed (also surfaced in-app via the ⚑ footer + inline note chips):

  • Spectrum Health link — per your instruction, no Spectrum ID is collected; we assume the back-end already holds the health record. Caregiver enters basic info only. Flagged as an open client question.
  • Service list & multi-select — services taken from the brief; the board tagged "how multi-select works" as R&D.
  • Confirmation method — board said "method TBD"; simulated as in-app push.
  • Nurse status feed — comes from the nurse API in the real build; simulated on a timer here.
  • Home hub + day/time picker — not their own stickies; added so the flow loops. ASAP/now and recurring visits were should-haves, left out.
  • One patient per caregiver in beta ("add another" is a placeholder).

Embedding

It's a full standalone document with <meta charset> and viewport set, styles scoped under #ch-app, and a phone frame that fills its container (min(412px, 94vw) × min(860px, 94vh)). Drop it in an iframe:

<iframe src="caring-health-prototype.html" style="width:100%;height:780px;border:0" title="Caring Health prototype"></iframe>

One real bug I caught and fixed during testing: the "visit in progress" card was leaking onto a fresh Home screen (a display:flex rule was overriding the hidden attribute) — fixed with a [hidden] guard.

Want me to fold in a basic visit summary screen after completion, or a phone-number login path? Both were on the board as should-haves and would be quick next steps if you want to show range beyond strict must-haves.
