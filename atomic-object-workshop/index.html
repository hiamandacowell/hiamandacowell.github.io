Identified gaps in Claude Code prompt and formulated targeted refinements ›

Your prompt already has good bones. The framing is clear, the assumptions are doing real work, and pointing Claude Code at the green "must have" stickies as the scope boundary is exactly the right move. A few targeted additions will change the output meaningfully, plus one real decision you should make before you run it.

The one decision to make first: what should Claude Code actually build it in?

You never name a stack, so Claude Code will pick one for you. Two sensible directions:

  • A single self-contained HTML file. Opens in any browser, no build step, trivial to send to someone or click through on your phone. Best if the point is "look at this flow" and move on.
  • A small React + Vite project. Cleaner component reuse across the repeated screens, easier to extend if this becomes a thing you keep iterating on. Slightly more setup.

For a one-shot artifact to evaluate a flow, I'd lean single-file HTML. If you think you'll keep building on it, React. Tell me which and I'll fold it in.

Surgical additions that will tighten the result:

A reading instruction up top, since two documents are involved:

Two files are attached: a printout of a Miro journey map and a redacted brief. Read both before writing code. The board export got split across pages, so the page breaks aren't meaningful. Reconstruct the intended flow across them. Pull beta scope from the green "must have" stickies only.

A fidelity line, so it doesn't over-invest in polish:

This is mid-fidelity, meant to test the flow and not the visual design. Clean, legible, in a mobile phone frame, with real tap-through navigation. Skip production assets and heavy styling.

A definition of done, so screens actually connect:

Done means: starting from Dana signing up, I can tap through every must-have screen and reach the core loop. No screen dead-ends unless it's an obvious placeholder.

A gap-handling instruction, which maps directly to your stated goal of deciding where research goes next:

Where the board or brief leaves something undefined, make a reasonable choice and surface it. Leave a short visible note on the screen (or a running list at the end) flagging where we guessed and where we'd need to go back to the user. Don't silently invent the parts we knew were open questions.

That last one turns your known gaps into a visible map of where research and design need to go, which is the actual reason you're building this.

One small thing in your existing text: the individual-flows clarification ("those were just for brainstorming, they have our names on them") is useful but currently mid-paragraph. Move it into the assumptions list as its own line so Claude Code treats it as a hard "ignore these" rule rather than narrative context.

The health-info assumption reads clearly as written, so I'd leave it.

Want me to assemble the full refined prompt once you've picked HTML vs React? I'll keep your voice and just slot these in rather than rewriting it.

A couple of notes on choices I made:

I put the reading instruction and the build/fidelity/done lines between your two opening paragraphs and the assumptions list, so Claude Code reads the context, then gets its marching orders, then the hard constraints. That ordering tends to keep it from skimming the constraints.

The embed detail is in the build line ("run on its own in a browser and on mobile"), but one thing to watch: a self-contained file is easy to embed via an iframe on your site, which keeps its styles from colliding with your page styles. If Claude Code instead gives you a fragment meant to drop straight into the page body, you'll want to ask it to wrap the whole thing so it's iframe-ready. Worth a glance when the output comes back.

What I built

caring-health-prototype.html — one self-contained file (HTML/CSS/JS, no build step, no external assets). I rendered all 41 board slices to reconstruct the journey map, read the brief and Dana's persona, and pulled scope only from the green "must have" stickies across the three activity bands. Named brainstorm flows, yellow should/could, and red "nopes" were excluded.

I tap-tested the whole path in a browser: Dana signs up → creates password → enters caregiver info → enters patient (Walter) info → home → request a visit → pick services → location → day/time → review → confirm → live nurse tracking → back home with the visit logged. No dead-ends; the core loop closes.

Where we guessed (also surfaced in-app via the ⚑ footer + inline note chips):

  • Spectrum Health link — per your instruction, no Spectrum ID is collected; we assume the back-end already holds the health record. Caregiver enters basic info only. Flagged as an open client question.
  • Service list & multi-select — services taken from the brief; the board tagged "how multi-select works" as R&D.
  • Confirmation method — board said "method TBD"; simulated as in-app push.
  • Nurse status feed — comes from the nurse API in the real build; simulated on a timer here.
  • Home hub + day/time picker — not their own stickies; added so the flow loops. ASAP/now and recurring visits were should-haves, left out.
  • One patient per caregiver in beta ("add another" is a placeholder).

One real bug I caught and fixed during testing: the "visit in progress" card was leaking onto a fresh Home screen (a display:flex rule was overriding the hidden attribute) — fixed with a [hidden] guard.
